30/*
31 Implementation notes:
32
33 - An OOP style is used where each "class" is made up of a struct and a number of functions prefixed with the struct name.
34
35 - struct fields that are const are readonly. Either they are set in a create function and can never be changed, or they can only
36 be changed by calling a function.
37
38 - Inheritance is done using a struct field named "super" as the first field, allowing the struct to be cast to its "super class".
39 This works because a pointer to a struct is guaranteed to be a pointer to the first struct field.
40
41 - Classes intended for inheritance provide init/deinit functions which subclasses must call in their create/dispose functions.
42
43 - Polymorphism is done by a base class providing function pointers in its init function. The public API delegates to these
44 function pointers.
45
46 - Subclasses do not provide a dispose function, instead the base class' dispose function should be used, which will delegate to
47 a dispose function pointer.
48
49 - Classes not designed for inheritance cannot be extended because they may use an internal subclass to hide private data and don't
50 expose function pointers.
51
52 - The public API hides implementation details, such as init/deinit functions. An internal API is exposed by extension.h to allow
53 classes to be extended. Internal functions begin with underscore (_).
54
55 - OOP in C tends to lose type safety. Macros for casting are provided in extension.h to give context for why a cast is being done.
56
57 */
