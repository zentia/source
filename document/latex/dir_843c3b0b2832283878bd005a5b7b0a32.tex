\doxysubsection{详细描述}
xx\+Hash is an Extremely fast Hash algorithm, processing at RAM speed limits. Code is highly portable, and produces hashes identical across all platforms (little / big endian). The library includes the following algorithms \+:
\begin{DoxyItemize}
\item XXH32 \+: generates 32-\/bit hashes, using 32-\/bit arithmetic
\item XXH64 \+: generates 64-\/bit hashes, using 64-\/bit arithmetic
\item XXH3 (since {\ttfamily v0.\+8.\+0})\+: generates 64 or 128-\/bit hashes, using vectorized arithmetic. The 128-\/bit variant is called XXH128.
\end{DoxyItemize}

All variants successfully complete the \href{https://code.google.com/p/smhasher/wiki/SMHasher}{\texttt{ SMHasher}} test suite which evaluates the quality of hash functions (collision, dispersion and randomness). Additional tests, which evaluate more thoroughly speed and collision properties of 64-\/bit hashes, \href{https://github.com/Cyan4973/xxHash/tree/dev/tests}{\texttt{ are also provided}}.

Performance on large data is only one part of the picture. Hashing is also very useful in constructions like hash tables and bloom filters. In these use cases, it\textquotesingle{}s frequent to hash a lot of small data (starting at a few bytes). Algorithm\textquotesingle{}s performance can be very different for such scenarios, since parts of the algorithm, such as initialization or finalization, become fixed cost. The impact of branch mis-\/prediction also becomes much more present.

Speed is not the only property that matters. Produced hash values must respect excellent dispersion and randomness properties, so that any sub-\/section of it can be used to maximally spread out a table or index, as well as reduce the amount of collisions to the minimal theoretical level, following the \href{https://en.wikipedia.org/wiki/Birthday_problem}{\texttt{ birthday paradox}}.

{\ttfamily xx\+Hash} has been tested with Austin Appleby\textquotesingle{}s excellent SMHasher test suite, and passes all tests, ensuring reasonable quality levels. It also passes extended tests from \href{https://github.com/rurban/smhasher}{\texttt{ newer forks of SMHasher}}, featuring additional scenarios and conditions.

Finally, xx\+Hash provides its own \href{https://github.com/Cyan4973/xxHash/tree/dev/tests/collisions}{\texttt{ massive collision tester}}, able to generate and compare billions of hashes to test the limits of 64-\/bit hash algorithms. On this front too, xx\+Hash features good results, in line with the \href{https://en.wikipedia.org/wiki/Birthday_problem}{\texttt{ birthday paradox}}. A more detailed analysis is documented \href{https://github.com/Cyan4973/xxHash/wiki/Collision-ratio-comparison}{\texttt{ in the wiki}}.\hypertarget{README.md_autotoc_md445}{}\doxysubsubsection{\texorpdfstring{Build modifiers}{Build modifiers}}\label{README.md_autotoc_md445}
The following macros can be set at compilation time to modify {\ttfamily libxxhash}\textquotesingle{}s behavior. They are generally disabled by default.


\begin{DoxyItemize}
\item {\ttfamily XXH\+\_\+\+INLINE\+\_\+\+ALL}\+: Make all functions {\ttfamily inline}, implementation is directly included within {\ttfamily \doxylink{xxhash_8h}{xxhash.\+h}}. Inlining functions is beneficial for speed, notably for small keys. It\textquotesingle{}s {\itshape extremely effective} when key\textquotesingle{}s length is expressed as {\itshape a compile time constant}, with performance improvements observed in the +200\% range . See \href{https://fastcompression.blogspot.com/2018/03/xxhash-for-small-keys-impressive-power.html}{\texttt{ this article}} for details.
\item {\ttfamily XXH\+\_\+\+PRIVATE\+\_\+\+API}\+: same outcome as {\ttfamily XXH\+\_\+\+INLINE\+\_\+\+ALL}. Still available for legacy support. The name underlines that {\ttfamily XXH\+\_\+\texorpdfstring{$\ast$}{*}} symbol names will not be exported.
\item {\ttfamily XXH\+\_\+\+STATIC\+\_\+\+LINKING\+\_\+\+ONLY}\+: gives access to internal state declaration, required for static allocation. Incompatible with dynamic linking, due to risks of ABI changes.
\item {\ttfamily XXH\+\_\+\+NAMESPACE}\+: Prefixes all symbols with the value of {\ttfamily XXH\+\_\+\+NAMESPACE}. This macro can only use compilable character set. Useful to evade symbol naming collisions, in case of multiple inclusions of xx\+Hash\textquotesingle{}s source code. Client applications still use the regular function names, as symbols are automatically translated through {\ttfamily \doxylink{xxhash_8h}{xxhash.\+h}}.
\item {\ttfamily XXH\+\_\+\+FORCE\+\_\+\+ALIGN\+\_\+\+CHECK}\+: Use a faster direct read path when input is aligned. This option can result in dramatic performance improvement on architectures unable to load memory from unaligned addresses when input to hash happens to be aligned on 32 or 64-\/bit boundaries. It is (slightly) detrimental on platform with good unaligned memory access performance (same instruction for both aligned and unaligned accesses). This option is automatically disabled on {\ttfamily x86}, {\ttfamily x64} and {\ttfamily aarch64}, and enabled on all other platforms.
\item {\ttfamily XXH\+\_\+\+FORCE\+\_\+\+MEMORY\+\_\+\+ACCESS}\+: The default method {\ttfamily 0} uses a portable {\ttfamily memcpy()} notation. Method {\ttfamily 1} uses a gcc-\/specific {\ttfamily packed} attribute, which can provide better performance for some targets. Method {\ttfamily 2} forces unaligned reads, which is not standard compliant, but might sometimes be the only way to extract better read performance. Method {\ttfamily 3} uses a byteshift operation, which is best for old compilers which don\textquotesingle{}t inline {\ttfamily memcpy()} or big-\/endian systems without a byteswap instruction.
\item {\ttfamily XXH\+\_\+\+CPU\+\_\+\+LITTLE\+\_\+\+ENDIAN}\+: By default, endianness is determined by a runtime test resolved at compile time. If, for some reason, the compiler cannot simplify the runtime test, it can cost performance. It\textquotesingle{}s possible to skip auto-\/detection and simply state that the architecture is little-\/endian by setting this macro to 1. Setting it to 0 states big-\/endian.
\item {\ttfamily XXH\+\_\+\+ENABLE\+\_\+\+AUTOVECTORIZE}\+: Auto-\/vectorization may be triggered for XXH32 and XXH64, depending on cpu vector capabilities and compiler version. Note\+: auto-\/vectorization tends to be triggered more easily with recent versions of {\ttfamily clang}. For XXH32, SSE4.\+1 or equivalent (NEON) is enough, while XXH64 requires AVX512. Unfortunately, auto-\/vectorization is generally detrimental to XXH performance. For this reason, the xxhash source code tries to prevent auto-\/vectorization by default. That being said, systems evolve, and this conclusion is not forthcoming. For example, it has been reported that recent Zen4 cpus are more likely to improve performance with vectorization. Therefore, should you prefer or want to test vectorized code, you can enable this flag\+: it will remove the no-\/vectorization protection code, thus making it more likely for XXH32 and XXH64 to be auto-\/vectorized.
\item {\ttfamily XXH32\+\_\+\+ENDJMP}\+: Switch multi-\/branch finalization stage of XXH32 by a single jump. This is generally undesirable for performance, especially when hashing inputs of random sizes. But depending on exact architecture and compiler, a jump might provide slightly better performance on small inputs. Disabled by default.
\item {\ttfamily XXH\+\_\+\+IMPORT}\+: MSVC specific\+: should only be defined for dynamic linking, as it prevents linkage errors.
\item {\ttfamily XXH\+\_\+\+NO\+\_\+\+STDLIB}\+: Disable invocation of {\ttfamily \texorpdfstring{$<$}{<}stdlib.\+h\texorpdfstring{$>$}{>}} functions, notably {\ttfamily \doxylink{mimalloc-override_8h_a99aa8fd9f4831e1b0fa9e1cdcde3e249}{malloc()}} and {\ttfamily \doxylink{mimalloc-override_8h_a9d4b5df3530d1bc733070a4669ba6ebc}{free()}}. {\ttfamily libxxhash}\textquotesingle{}s {\ttfamily XXH\texorpdfstring{$\ast$}{*}\+\_\+create\+State()} will always fail and return {\ttfamily NULL}. But one-\/shot hashing (like {\ttfamily \doxylink{group___x_x_h32__family_ga49360c30d0eef2cd4e20cabfe6ab1c55}{XXH32()}}) or streaming using statically allocated states still work as expected. This build flag is useful for embedded environments without dynamic allocation.
\item {\ttfamily XXH\+\_\+memcpy}, {\ttfamily XXH\+\_\+memset}, {\ttfamily XXH\+\_\+memcmp} \+: redirect {\ttfamily memcpy()}, {\ttfamily memset()} and {\ttfamily memcmp()} to some user-\/selected symbol at compile time. Redirecting all 3 removes the need to include {\ttfamily \texorpdfstring{$<$}{<}\doxylink{string_8h}{string.\+h}\texorpdfstring{$>$}{>}} standard library.
\item {\ttfamily XXH\+\_\+\+NO\+\_\+\+EXTERNC\+\_\+\+GUARD}\+: When {\ttfamily \doxylink{xxhash_8h}{xxhash.\+h}} is compiled in C++ mode, removes the {\ttfamily extern "{}\+C"{} \{ .. \}} block guard.
\item {\ttfamily XXH\+\_\+\+DEBUGLEVEL} \+: When set to any value \texorpdfstring{$>$}{>}= 1, enables {\ttfamily assert()} statements. This (slightly) slows down execution, but may help finding bugs during debugging sessions.
\end{DoxyItemize}\hypertarget{README.md_autotoc_md446}{}\doxysubsubsubsection{\texorpdfstring{Binary size control}{Binary size control}}\label{README.md_autotoc_md446}

\begin{DoxyItemize}
\item {\ttfamily XXH\+\_\+\+NO\+\_\+\+XXH3} \+: removes symbols related to {\ttfamily XXH3} (both 64 \& 128 bits) from generated binary. {\ttfamily XXH3} is by far the largest contributor to {\ttfamily libxxhash} size, so it\textquotesingle{}s useful to reduce binary size for applications which do not employ {\ttfamily XXH3}.
\item {\ttfamily XXH\+\_\+\+NO\+\_\+\+LONG\+\_\+\+LONG}\+: removes compilation of algorithms relying on 64-\/bit {\ttfamily long long} types which include {\ttfamily XXH3} and {\ttfamily XXH64}. Only {\ttfamily XXH32} will be compiled. Useful for targets (architectures and compilers) without 64-\/bit support.
\item {\ttfamily XXH\+\_\+\+NO\+\_\+\+STREAM}\+: Disables the streaming API, limiting the library to single shot variants only.
\item {\ttfamily XXH\+\_\+\+NO\+\_\+\+INLINE\+\_\+\+HINTS}\+: By default, xx\+Hash uses {\ttfamily \+\_\+\+\_\+attribute\+\_\+\+\_\+((always\+\_\+inline))} and {\ttfamily \+\_\+\+\_\+forceinline} to improve performance at the cost of code size. Defining this macro to 1 will mark all internal functions as {\ttfamily static}, allowing the compiler to decide whether to inline a function or not. This is very useful when optimizing for smallest binary size, and is automatically defined when compiling with {\ttfamily -\/O0}, {\ttfamily -\/Os}, {\ttfamily -\/Oz}, or {\ttfamily -\/fno-\/inline} on GCC and Clang. It may also be required to successfully compile using {\ttfamily -\/Og}, depending on compiler version.
\item {\ttfamily XXH\+\_\+\+SIZE\+\_\+\+OPT}\+: {\ttfamily 0}\+: default, optimize for speed {\ttfamily 1}\+: default for {\ttfamily -\/Os} and {\ttfamily -\/Oz}\+: disables some speed hacks for size optimization {\ttfamily 2}\+: makes code as small as possible, performance may cry
\end{DoxyItemize}\hypertarget{README.md_autotoc_md447}{}\doxysubsubsubsection{\texorpdfstring{Build modifiers specific for XXH3}{Build modifiers specific for XXH3}}\label{README.md_autotoc_md447}

\begin{DoxyItemize}
\item {\ttfamily XXH\+\_\+\+VECTOR} \+: manually select a vector instruction set (default\+: auto-\/selected at compilation time). Available instruction sets are {\ttfamily XXH\+\_\+\+SCALAR}, {\ttfamily XXH\+\_\+\+SSE2}, {\ttfamily XXH\+\_\+\+AVX2}, {\ttfamily XXH\+\_\+\+AVX512}, {\ttfamily XXH\+\_\+\+NEON} and {\ttfamily XXH\+\_\+\+VSX}. Compiler may require additional flags to ensure proper support (for example, {\ttfamily gcc} on x86\+\_\+64 requires {\ttfamily -\/mavx2} for {\ttfamily AVX2}, or {\ttfamily -\/mavx512f} for {\ttfamily AVX512}).
\item {\ttfamily XXH\+\_\+\+PREFETCH\+\_\+\+DIST} \+: select prefetching distance. For close-\/to-\/metal adaptation to specific hardware platforms. XXH3 only.
\item {\ttfamily XXH\+\_\+\+NO\+\_\+\+PREFETCH} \+: disable prefetching. Some platforms or situations may perform better without prefetching. XXH3 only.
\end{DoxyItemize}\hypertarget{README.md_autotoc_md448}{}\doxysubsubsubsection{\texorpdfstring{Makefile variables}{Makefile variables}}\label{README.md_autotoc_md448}
When compiling the Command Line Interface {\ttfamily xxhsum} using {\ttfamily make}, the following environment variables can also be set \+:
\begin{DoxyItemize}
\item {\ttfamily DISPATCH=1} \+: use {\ttfamily \doxylink{xxh__x86dispatch_8c}{xxh\+\_\+x86dispatch.\+c}}, select at runtime between {\ttfamily scalar}, {\ttfamily sse2}, {\ttfamily avx2} or {\ttfamily avx512} instruction set. \item {\ttfamily LIBXXH\+\_\+\+DISPATCH=1} \+: same idea, implemented a runtime vector extension detector, but within {\ttfamily libxxhash}. This parameter is disabled by default. When enabled (only valid for {\ttfamily x86}/{\ttfamily x64} systems), new symbols published in {\ttfamily \doxylink{xxh__x86dispatch_8h}{xxh\+\_\+x86dispatch.\+h}} become accessible. At the time of this writing, it\textquotesingle{}s required to include {\ttfamily \doxylink{xxh__x86dispatch_8h}{xxh\+\_\+x86dispatch.\+h}} in order to access the symbols with runtime vector extension detection.
\item {\ttfamily XXH\+\_\+1\+ST\+\_\+\+SPEED\+\_\+\+TARGET} \+: select an initial speed target, expressed in MB/s, for the first speed test in benchmark mode. Benchmark will adjust the target at subsequent iterations, but the first test is made "{}blindly"{} by targeting this speed. Currently conservatively set to 10 MB/s, to support very slow (emulated) platforms.
\item {\ttfamily NODE\+\_\+\+JS=1} \+: When compiling {\ttfamily xxhsum} for Node.\+js with Emscripten, this links the {\ttfamily NODERAWFS} library for unrestricted filesystem access and patches {\ttfamily isatty} to make the command line utility correctly detect the terminal. This does make the binary specific to Node.\+js.
