\subsubsection*{template$<$typename GC, typename Key, typename \mbox{\hyperlink{struct_value}{Value}}, typename Traits$>$\newline
class cds\+::container\+::\+Iterable\+KVList$<$ GC, Key, Value, Traits $>$}
Iterable ordered list for key-\/value pair 

\label{classcds_1_1container_1_1_iterable_k_v_list_cds_nonintrusive_IterableKVList_gc}%
\Hypertarget{classcds_1_1container_1_1_iterable_k_v_list_cds_nonintrusive_IterableKVList_gc}%
 This is key-\/value variation of non-\/intrusive {\ttfamily \doxylink{classcds_1_1container_1_1_iterable_list}{Iterable\+List}}. Like standard container, this implementation split a value stored into two part -\/ constant key and alterable value.

Usually, ordered single-\/linked list is used as a building block for the hash table implementation. Iterable list is suitable for almost append-\/only hash table because the list doesn\textquotesingle{}t delete its internal node when erasing a key but it is marked them as empty to be reused in the future. However, plenty of empty nodes degrades performance.

The complexity of searching is {\ttfamily O(\+N)}.

Template arguments\+:
\begin{DoxyItemize}
\item {\ttfamily GC} -\/ garbage collector used
\item {\ttfamily Key} -\/ key type of an item stored in the list. It should be copy-\/constructible
\item {\ttfamily \doxylink{struct_value}{Value}} -\/ value type stored in a list
\item {\ttfamily Traits} -\/ type traits, default is {\ttfamily \doxylink{structcds_1_1container_1_1iterable__list_1_1traits}{iterable\+\_\+list\+::traits}}
