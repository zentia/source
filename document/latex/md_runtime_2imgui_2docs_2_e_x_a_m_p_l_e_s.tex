{\bfseries{About mouse cursor latency}}

Dear \doxylink{namespace_im_gui}{Im\+Gui} does not introduce significant extra lag for most behaviors, e.\+g. the last value passed to \textquotesingle{}io.\+Add\+Mouse\+Pos\+Event()\textquotesingle{} before New\+Frame() will result in windows being moved to the right spot at the time of End\+Frame()/\+Render(). At 60 FPS your experience should be pleasant.

However, consider that OS mouse cursors are typically rendered through a very specific hardware-\/accelerated path, which makes them feel smoother than the majority of content rendered via regular graphics API (including, but not limited to Dear \doxylink{namespace_im_gui}{Im\+Gui} windows). Because UI rendering and interaction happens on the same plane as the mouse, that disconnect may be jarring to particularly sensitive users. You may experiment with enabling the io.\+Mouse\+Draw\+Cursor flag to request Dear \doxylink{namespace_im_gui}{Im\+Gui} to draw a mouse cursor using the regular graphics API, to help you visualize the difference between a "{}hardware"{} cursor and a regularly rendered software cursor. However, rendering a mouse cursor at 60 FPS will feel sluggish, so you likely won\textquotesingle{}t want to enable that at all times. It might be beneficial for the user experience to switch to a software rendered cursor {\itshape only} when an interactive drag is in progress.

Note that some setup configurations or GPU drivers may introduce additional display lag depending on their settings. If you notice that dragging windows is laggy and you are not sure what the cause is\+: try drawing a simple 2D shape directly under the mouse cursor to help identify the issue!
