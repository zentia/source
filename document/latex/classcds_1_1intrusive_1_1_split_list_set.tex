Split-\/ordered list 

\label{classcds_1_1intrusive_1_1_split_list_set_cds_intrusive_SplitListSet_hp}%
\Hypertarget{classcds_1_1intrusive_1_1_split_list_set_cds_intrusive_SplitListSet_hp}%
 \doxylink{struct_hash}{Hash} table implementation based on split-\/ordered list algorithm discovered by Ori Shalev and Nir Shavit, see
\begin{DoxyItemize}
\item \mbox{[}2003\mbox{]} Ori Shalev, Nir Shavit "{}\+Split-\/\+Ordered Lists -\/ Lock-\/free Resizable Hash Tables"{}
\item \mbox{[}2008\mbox{]} Nir Shavit "{}\+The Art of Multiprocessor Programming"{}
\end{DoxyItemize}

The split-\/ordered list is a lock-\/free implementation of an extensible unbounded hash table. It uses original recursive split-\/ordering algorithm discovered by Ori Shalev and Nir Shavit that allows to split buckets without item moving on resizing.

\label{classcds_1_1intrusive_1_1_split_list_set_cds_SplitList_algo_desc}%
\Hypertarget{classcds_1_1intrusive_1_1_split_list_set_cds_SplitList_algo_desc}%
{\bfseries{Short description}} \mbox{[}from \mbox{[}2003\mbox{]} Ori Shalev, Nir Shavit "{}\+Split-\/\+Ordered Lists -\/ Lock-\/free Resizable Hash Tables"{}\mbox{]}

The algorithm keeps all the items in one lock-\/free linked list, and gradually assigns the bucket pointers to the places in the list where a sublist of \textquotesingle{}correct\textquotesingle{} items can be found. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} bucket is initialized upon first access by assigning it to a new \textquotesingle{}dummy\textquotesingle{} node (dashed contour) in the list, preceding all items that should be in that bucket. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} newly created bucket splits an older bucket\textquotesingle{}s chain, reducing the access cost to its items. The table uses a modulo 2\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}i hash (there are known techniques for \textquotesingle{}pre-\/hashing\textquotesingle{} before a modulo 2\texorpdfstring{$\ast$}{*}\texorpdfstring{$\ast$}{*}i hash to overcome possible binary correlations among values). The table starts at size 2 and repeatedly doubles in size.

Unlike moving an item, the operation of directing a bucket pointer can be done in a single CAS operation, and since items are not moved, they are never \textquotesingle{}lost\textquotesingle{}. However, to make this approach work, one must be able to keep the items in the list sorted in such a way that any bucket\textquotesingle{}s sublist can be \textquotesingle{}split\textquotesingle{} by directing a new bucket pointer within it. This operation must be recursively repeatable, as every split bucket may be split again and again as the hash table grows. To achieve this goal the authors introduced recursive split-\/ordering, a new ordering on keys that keeps items in a given bucket adjacent in the list throughout the repeated splitting process.

Magically, yet perhaps not surprisingly, recursive split-\/ordering is achieved by simple binary reversal\+: reversing the bits of the hash key so that the new key\textquotesingle{}s most significant bits (MSB) are those that were originally its least significant. The split-\/order keys of regular nodes are exactly the bit-\/reverse image of the original keys after turning on their MSB. For example, items 9 and 13 are in the {\ttfamily 1 mod 4} bucket, which can be recursively split in two by inserting a new node between them.

To insert (respectively delete or search for) an item in the hash table, hash its key to the appropriate bucket using recursive split-\/ordering, follow the pointer to the appropriate location in the sorted items list, and traverse the list until the key\textquotesingle{}s proper location in the split-\/ordering (respectively until the key or a key indicating the item is not in the list is found). Because of the combinatorial structure induced by the split-\/ordering, this will require traversal of no more than an expected constant number of items.

The design is modular\+: to implement the ordered items list, you can use one of several non-\/blocking list-\/based set algorithms\+: \doxylink{classcds_1_1intrusive_1_1_michael_list}{Michael\+List}, \doxylink{classcds_1_1intrusive_1_1_lazy_list}{Lazy\+List}.

Forward iterator 

The forward iterator is based on {\ttfamily Ordered\+List} forward iterator and has some features\+:
\begin{DoxyItemize}
\item it has no post-\/increment operator
\item it iterates items in unordered fashion
\item iterator cannot be moved across thread boundary because it may contain GC\textquotesingle{}s guard that is thread-\/private GC data.
\end{DoxyItemize}

Iterator thread safety depends on type of {\ttfamily Ordered\+List\+:} 
\begin{DoxyItemize}
\item for {\ttfamily \doxylink{classcds_1_1intrusive_1_1_michael_list}{Michael\+List}} and {\ttfamily \doxylink{classcds_1_1intrusive_1_1_lazy_list}{Lazy\+List}\+:} iterator guarantees safety even if you delete the item that iterator points to because that item is guarded by hazard pointer. However, in case of concurrent deleting operations it is no guarantee that you iterate all item in the set. Moreover, a crash is possible when you try to iterate the next element that has been deleted by concurrent thread. Use this iterator on the concurrent container for debugging purpose only.
\item for {\ttfamily \doxylink{classcds_1_1intrusive_1_1_iterable_list}{Iterable\+List}\+:} iterator is thread-\/safe. You may use it freely in concurrent environment. 
\end{DoxyItemize}\Hypertarget{classcds_1_1intrusive_1_1_split_list_set_ac4ac7447e2749a53db596bef99378f0a}\index{cds::intrusive::SplitListSet$<$ GC, OrderedList, Traits $>$@{cds::intrusive::SplitListSet$<$ GC, OrderedList, Traits $>$}!key\_comparator@{key\_comparator}}
