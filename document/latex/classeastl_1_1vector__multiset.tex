\subsubsection*{template$<$typename Key, typename \mbox{\hyperlink{namespaceeastl_ae3d33fc51c4c15a85aee95335a5d336d}{Compare}} = eastl\+::less$<$\+Key$>$, typename Allocator = EASTLAllocator\+Type, typename Random\+Access\+Container = eastl\+::vector$<$\+Key, Allocator$>$$>$\newline
class eastl\+::vector\+\_\+multiset$<$ Key, Compare, Allocator, Random\+Access\+Container $>$}
\doxylink{classeastl_1_1vector__multiset}{vector\+\_\+multiset}

Implements a multiset via a random access container such as a vector. This container is also known as a sorted\+\_\+vector. We choose to call it \doxylink{classeastl_1_1vector__multiset}{vector\+\_\+multiset}, as that is a more consistent universally applicable name for it in this library.

Note that with \doxylink{classeastl_1_1vector__set}{vector\+\_\+set}, \doxylink{classeastl_1_1vector__multiset}{vector\+\_\+multiset}, \doxylink{classeastl_1_1vector__map}{vector\+\_\+map}, \doxylink{classeastl_1_1vector__multimap}{vector\+\_\+multimap} that the modification of the container potentially invalidates all existing iterators into the container, unlike what happens with conventional sets and maps.

This type could conceptually use a \doxylink{structeastl_1_1array}{eastl\+::array} as its underlying container, however the current design requires an allocator aware container. Consider using a \doxylink{classeastl_1_1fixed__vector}{fixed\+\_\+vector} instead.

To consider\+: std\+::multiset has the limitation that values in the set cannot be modified, with the idea that modifying them would change their sort order. We have the opportunity to make it so that values can be modified via changing iterators to be non-\/const, with the downside being that the container can get screwed up if the user screws up. Alternatively, we can do what std STL does and require the user to make their stored classes use \textquotesingle{}mutable\textquotesingle{} as needed. See the C++ standard defect report \#103 (DR 103) for a discussion of this.
