\doxysection{Limit the Maximum Concurrency}
\hypertarget{_limit_the_maximum_concurrency}{}\label{_limit_the_maximum_concurrency}\index{Limit the Maximum Concurrency@{Limit the Maximum Concurrency}}
This chapters discusses how to limit the concurrency or the maximum number of workers in your Taskflow applications.\hypertarget{_limit_the_maximum_concurrency_DefineASemaphore}{}\doxysubsection{\texorpdfstring{Define a Semaphore}{Define a Semaphore}}\label{_limit_the_maximum_concurrency_DefineASemaphore}
Taskflow provides a mechanism, \doxylink{classtf_1_1_semaphore}{tf\+::\+Semaphore}, for you to limit the maximum concurrency in a section of tasks. You can let a task acquire/release one or multiple semaphores before/after executing its work. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} task can acquire and release a semaphore, or just acquire or just release it. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} \doxylink{classtf_1_1_semaphore}{tf\+::\+Semaphore} object starts with an initial value. As long as that value is above 0, tasks can acquire the semaphore and do their work. If the value is 0 or less, a task trying to acquire the semaphore will not run but goes to a waiting list of that semaphore. When the semaphore is released by another task, it reschedules all tasks on that waiting list.

Without semaphores, each pair of tasks, e.\+g., {\ttfamily from-\/0 -\/\texorpdfstring{$>$}{>} to-\/0}, will run independently and concurrently. However, the program forces each {\ttfamily from} task to acquire the semaphore before running its work and not to release it until its paired {\ttfamily to} task is done. This constraint forces each pair of tasks to run sequentially, while the order of which pair runs first is up to the scheduler.\hypertarget{_limit_the_maximum_concurrency_UseSemaphoresAcrossDifferentTasks}{}\doxysubsection{\texorpdfstring{Use Semaphores Across Different Tasks}{Use Semaphores Across Different Tasks}}\label{_limit_the_maximum_concurrency_UseSemaphoresAcrossDifferentTasks}
You can use semaphores to limit the concurrency across different sections of taskflow graphs. When you submit multiple taskflows to an executor, the executor view them as a bag of dependent tasks. It does not matter which task in which taskflow graph acquires or releases a semaphore.

The above examples creates one task from each taskflow and submits the two taskflows to the executor. Again, under normal circumstances, the two tasks can run concurrently, but the semaphore restricts one worker to run the two task sequentially in arbitrary order.\hypertarget{_limit_the_maximum_concurrency_DefineAConflictGraph}{}\doxysubsection{\texorpdfstring{Define a Conflict Graph}{Define a Conflict Graph}}\label{_limit_the_maximum_concurrency_DefineAConflictGraph}
One important application of \doxylink{classtf_1_1_semaphore}{tf\+::\+Semaphore} is {\itshape conflict-\/aware scheduling} using a conflict graph. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} conflict graph is a {\itshape undirected} graph where each vertex represents a task and each edge represents a conflict between a pair of tasks. When a task conflicts with another task, they cannot run together. Consider the conflict graph below, task {\ttfamily \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A}} conflicts with task {\ttfamily \doxylink{bench__gemm_8cpp_a37a83060ac796961b44991c836f083f7}{B}} and task {\ttfamily C} (and vice versa), meaning that {\ttfamily \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A}} cannot run together with {\ttfamily \doxylink{bench__gemm_8cpp_a37a83060ac796961b44991c836f083f7}{B}} and {\ttfamily C} whereas {\ttfamily \doxylink{bench__gemm_8cpp_a37a83060ac796961b44991c836f083f7}{B}} and {\ttfamily C} can run together.

We can create one semaphore of one concurrency for each edge in the conflict graph and let the two tasks of that edge acquire the semaphore. This organization forces the two tasks to not run concurrently.

Currently, \doxylink{classtf_1_1_semaphore}{tf\+::\+Semaphore} has limited support for exception handling and taskflow cancellation. If a task throws an exception or the taskflow is canceled, subsequent acquire and release operations on the semaphore may result in undefined behavior. To ensure correct behavior, you should call \doxylink{classtf_1_1_semaphore_ad0f7801055550b20b8c2ae6d6099f220}{tf\+::\+Semaphore\+::reset} before reusing the semaphore in the next run. For instance, in the code below, when task {\ttfamily \doxylink{bench__gemm_8cpp_a37a83060ac796961b44991c836f083f7}{B}} throws an exception, the executor will cancel the execution of the taskflow. That is, tasks {\ttfamily C} and {\ttfamily D} will not run, and thus no task will release the acquired semaphore. To resolve this situation, we must reset the semaphore to a clean state for the next run.
