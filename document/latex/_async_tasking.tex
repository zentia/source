Unlike std\+::async, the future object returned from \doxylink{classtf_1_1_executor_af960048056f7c6b5bc71f4f526f05df7}{tf\+::\+Executor\+::async} does not block on destruction until completing the function.

Launching asynchronous tasks from an executor is {\itshape thread-\/safe} and can be called by multiple threads both inside (i.\+e., worker) and outside the executor. Our scheduler autonomously detects whether an asynchronous task is submitted from an external thread or a worker thread and schedules its execution using work stealing.

By default, \doxylink{classtf_1_1_runtime}{tf\+::\+Runtime} does not join like \doxylink{classtf_1_1_subflow}{tf\+::\+Subflow}. All pending asynchronous tasks spawned by \doxylink{classtf_1_1_runtime}{tf\+::\+Runtime} are no longer controllable when their parent runtime disappears. It is your responsibility to properly synchronize spawned asynchronous tasks using \doxylink{classtf_1_1_runtime_afcc18484a95fd2a834940d878eaf4dfc}{tf\+::\+Runtime\+::corun\+\_\+all}.

\begin{DoxyAttention}{注意}
Creating asynchronous tasks from a runtime allows users to efficiently implement parallel algorithms using recursion, such as parallel sort (\doxylink{classtf_1_1_flow_builder_a35e180eb63de6c9f28e43185e837a4fa}{tf\+::\+Taskflow\+::sort}), that demands dynamic parallelism at runtime. 
\end{DoxyAttention}
