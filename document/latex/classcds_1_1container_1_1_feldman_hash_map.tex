\doxylink{struct_hash}{Hash} map based on multi-\/level array 

\label{classcds_1_1container_1_1_feldman_hash_map_cds_container_FeldmanHashMap_hp}%
\Hypertarget{classcds_1_1container_1_1_feldman_hash_map_cds_container_FeldmanHashMap_hp}%
 Source\+:
\begin{DoxyItemize}
\item \mbox{[}2013\mbox{]} Steven Feldman, Pierre La\+Borde, Damian Dechev "{}\+Concurrent Multi-\/level Arrays\+:          Wait-\/free Extensible Hash Maps"{}
\end{DoxyItemize}

\mbox{[}From the paper\mbox{]} The hardest problem encountered while developing a parallel hash map is how to perform a global resize, the process of redistributing the elements in a hash map that occurs when adding new buckets. The negative impact of blocking synchronization is multiplied during a global resize, because all threads will be forced to wait on the thread that is performing the involved process of resizing the hash map and redistributing the elements. {\ttfamily Feldman\+Hash\+Set} implementation avoids global resizes through new array allocation. By allowing concurrent expansion this structure is free from the overhead of an explicit resize, which facilitates concurrent operations.

The presented design includes dynamic hashing, the use of sub-\/arrays within the hash map data structure; which, in combination with {\bfseries{perfect hashing}}, means that each element has a unique final, as well as current, position. It is important to note that the perfect hash function required by our hash map is trivial to realize as any hash function that permutes the bits of the key is suitable. This is possible because of our approach to the hash function; we require that it produces hash values that are equal in size to that of the key. We know that if we expand the hash map a fixed number of times there can be no collision as duplicate keys are not provided for in the standard semantics of a hash map.

{\ttfamily Feldman\+Hash\+Map} is a multi-\/level array which has an internal structure similar to a tree\+:  The multi-\/level array differs from a tree in that each position on the tree could hold an array of nodes or a single node. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} position that holds a single node is a {\ttfamily data\+Node} which holds the hash value of a key and the value that is associated with that key; it is a simple struct holding two variables. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} {\ttfamily data\+Node} in the multi-\/level array could be marked. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} {\ttfamily marked\+Data\+Node} refers to a pointer to a {\ttfamily data\+Node} that has been bitmarked at the least significant bit (LSB) of the pointer to the node. This signifies that this {\ttfamily data\+Node} is contended. An expansion must occur at this node; any thread that sees this {\ttfamily marked\+Data\+Node} will try to replace it with an {\ttfamily array\+Node}; which is a position that holds an array of nodes. The pointer to an {\ttfamily array\+Node} is differentiated from that of a pointer to a {\ttfamily data\+Node} by a bitmark on the second-\/least significant bit.

{\ttfamily Feldman\+Hash\+Map} multi-\/level array is similar to a tree in that we keep a pointer to the root, which is a memory array called {\ttfamily head}. The length of the {\ttfamily head} memory array is unique, whereas every other {\ttfamily array\+Node} has a uniform length; a normal {\ttfamily array\+Node} has a fixed power-\/of-\/two length equal to the binary logarithm of a variable called {\ttfamily array\+Length}. The maximum depth of the tree, {\ttfamily max\+Depth}, is the maximum number of pointers that must be followed to reach any node. We define {\ttfamily current\+Depth} as the number of memory arrays that we need to traverse to reach the {\ttfamily array\+Node} on which we need to operate; this is initially one, because of {\ttfamily head}.

That approach to the structure of the hash map uses an extensible hashing scheme; {\bfseries{ the hash value is treated as a bit string}} and rehash incrementally.

\begin{DoxyNote}{注解}
Two important things you should keep in mind when you\textquotesingle{}re using {\ttfamily Feldman\+Hash\+Map}\+:
\begin{DoxyItemize}
\item all keys is converted to fixed-\/size bit-\/string by hash functor provided. You can use variable-\/length keys, for example, {\ttfamily std\+::string} as a key for {\ttfamily Feldman\+Hash\+Map}, but real key in the map will be fixed-\/size hash values of your keys. For the strings you may use well-\/known hashing algorithms like \href{https://en.wikipedia.org/wiki/Secure_Hash_Algorithm}{\texttt{ SHA1, SHA2}}, \href{https://en.wikipedia.org/wiki/MurmurHash}{\texttt{ Murmur\+Hash}}, \href{https://en.wikipedia.org/wiki/CityHash}{\texttt{ City\+Hash}} or its successor \href{https://code.google.com/p/farmhash/}{\texttt{ Farm\+Hash}} and so on, which converts variable-\/length strings to fixed-\/length bit-\/strings, and such hash values will be the keys in {\ttfamily Feldman\+Hash\+Map}. If your key is fixed-\/sized the hash functor is optional, see {\ttfamily \doxylink{structcds_1_1container_1_1feldman__hashmap_1_1traits_a6fd68f42f98b81c6b74bf2e86462f0a2}{feldman\+\_\+hashmap\+::traits\+::hash}} for explanation and examples.
\item {\ttfamily Feldman\+Hash\+Map} uses a perfect hashing. It means that if two different keys, for example, of type {\ttfamily std\+::string}, have identical hash then you cannot insert both that keys in the map. {\ttfamily Feldman\+Hash\+Map} does not maintain the key, it maintains its fixed-\/size hash value.
\end{DoxyItemize}
\end{DoxyNote}
The map supports \doxylink{classcds_1_1container_1_1_feldman_hash_map_cds_container_FeldmanHashMap_iterators}{bidirectional thread-\/safe iterators}.

Returns an iterator to the beginning of the map 

\label{classcds_1_1container_1_1_feldman_hash_map_cds_container_FeldmanHashMap_iterators}%
\Hypertarget{classcds_1_1container_1_1_feldman_hash_map_cds_container_FeldmanHashMap_iterators}%
The map supports thread-\/safe iterators\+: you may iterate over the map in multi-\/threaded environment. It is guaranteed that the iterators will remain valid even if another thread deletes the node the iterator points to\+: Hazard Pointer embedded into the iterator object protects the node from physical reclamation.

\begin{DoxyNote}{注解}
Since the iterator object contains hazard pointer that is a thread-\/local resource, the iterator should not be passed to another thread.
\end{DoxyNote}
Each iterator object supports the common interface\+:
\begin{DoxyItemize}
\item dereference operators\+: 
\begin{DoxyCode}{0}
\DoxyCodeLine{\mbox{\hyperlink{classcds_1_1container_1_1_feldman_hash_map_a9789fa7fd38e1567be79ea9a25fa6a54}{value\_type}}\ [\textcolor{keyword}{const}]\ *\ operator\ -\/>()\ noexcept}
\DoxyCodeLine{\mbox{\hyperlink{classcds_1_1container_1_1_feldman_hash_map_a9789fa7fd38e1567be79ea9a25fa6a54}{value\_type}}\ [const]\ \&\ operator\ *()\ noexcept}

\end{DoxyCode}

\item pre-\/increment and pre-\/decrement. Post-\/operators is not supported
\item equality operators {\ttfamily ==} and {\ttfamily !=}. Iterators are equal iff they point to the same cell of the same array node. Note that for two iterators {\ttfamily it1} and {\ttfamily it2}, the conditon {\ttfamily  it1 == it2 } does not entail {\ttfamily  \&(\texorpdfstring{$\ast$}{*}it1) == \&(\texorpdfstring{$\ast$}{*}it2) }
\item helper member function {\ttfamily release()} that clears internal hazard pointer. After {\ttfamily release()} the iterator points to {\ttfamily nullptr} but it still remain valid\+: further iterating is possible.
\end{DoxyItemize}

During iteration you may safely erase any item from the set; \doxylink{classcds_1_1container_1_1_feldman_hash_map_a414c6d3ba2b5b5ce60fe13058da9bdc0}{erase\+\_\+at()} function call doesn\textquotesingle{}t invalidate any iterator. If some iterator points to the item to be erased, that item is not deleted immediately but only after that iterator will be advanced forward or backward.

\begin{DoxyNote}{注解}
It is possible the item can be iterated more that once, for example, if an iterator points to the item in array node that is being splitted. 
\end{DoxyNote}
