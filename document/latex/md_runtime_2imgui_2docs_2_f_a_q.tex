\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md138}{}\doxysubsubsection{\texorpdfstring{Q. What is this library called?}{Q. What is this library called?}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md138}
{\bfseries{This library is called Dear \doxylink{namespace_im_gui}{Im\+Gui}}}. Please refer to it as Dear \doxylink{namespace_im_gui}{Im\+Gui} (not \doxylink{namespace_im_gui}{Im\+Gui}, not IMGUI).

(The library misleadingly started its life in 2014 as "{}\+Im\+Gui"{} due to the fact that I didn\textquotesingle{}t give it a proper name when I released 1.\+0, and had no particular expectation that it would take off. However, the term IMGUI (immediate-\/mode graphical user interface) was coined before and is being used in variety of other situations e.\+g. Unity uses it own implementation of the IMGUI paradigm. To reduce the ambiguity without affecting existing code bases, I have decided in December 2015 a fully qualified name "{}\+Dear Im\+Gui"{} for this library.\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md139}{}\doxysubsubsubsubsection{\texorpdfstring{\protect\doxysectlink{index}{Return to Index}{0}}{{Return to Index}}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md139}

\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md148}{}\doxysubsubsection{\texorpdfstring{Q\+: How can I tell whether to dispatch mouse/keyboard to Dear Im\+Gui or my application?}{Q\+: How can I tell whether to dispatch mouse/keyboard to Dear Im\+Gui or my application?}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md148}
You can read the {\ttfamily io.\+Want\+Capture\+Mouse}, {\ttfamily io.\+Want\+Capture\+Keyboard} and {\ttfamily io.\+Want\+Text\+Input} flags from the \doxylink{struct_im_gui_i_o}{Im\+Gui\+IO} structure.
\begin{DoxyItemize}
\item When {\ttfamily io.\+Want\+Capture\+Mouse} is set, you need to discard/hide the mouse inputs from your underlying application.
\item When {\ttfamily io.\+Want\+Capture\+Keyboard} is set, you need to discard/hide the keyboard inputs from your underlying application.
\item When {\ttfamily io.\+Want\+Text\+Input} is set, you can notify your OS/engine to popup an on-\/screen keyboard, if available (e.\+g. on a mobile phone, or console OS).
\end{DoxyItemize}

Important\+: you should always pass your mouse/keyboard inputs to Dear \doxylink{namespace_im_gui}{Im\+Gui}, regardless of the value {\ttfamily io.\+Want\+Capture\+Mouse}/{\ttfamily io.\+Want\+Capture\+Keyboard}. This is because e.\+g. we need to detect that you clicked in the void to unfocus its own windows, and other reasons.

\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md151}{}\doxysubsubsection{\texorpdfstring{Q\+: How can I enable keyboard or gamepad controls?}{Q\+: How can I enable keyboard or gamepad controls?}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md151}

\begin{DoxyItemize}
\item The gamepad/keyboard navigation is fairly functional and keeps being improved. The initial focus was to support game controllers, but keyboard is becoming increasingly and decently usable. Gamepad support is particularly useful to use Dear \doxylink{namespace_im_gui}{Im\+Gui} on a game console (e.\+g. PS4, Switch, XB1) without a mouse connected!
\item Keyboard\+: set {\ttfamily io.\+Config\+Flags \texorpdfstring{$\vert$}{|}= Im\+Gui\+Config\+Flags\+\_\+\+Nav\+Enable\+Keyboard} to enable.
\item Gamepad\+: set {\ttfamily io.\+Config\+Flags \texorpdfstring{$\vert$}{|}= Im\+Gui\+Config\+Flags\+\_\+\+Nav\+Enable\+Gamepad} to enable (with a supporting backend).
\item See \href{https://www.dearimgui.com/controls_sheets}{\texttt{ Control Sheets for Gamepads}} (reference PNG/\+PSD for PS4, XB1, Switch gamepads).
\item See {\ttfamily USING GAMEPAD/\+KEYBOARD NAVIGATION CONTROLS} section of \href{https://github.com/ocornut/imgui/blob/master/imgui.cpp}{\texttt{ imgui.\+cpp}} for more details.

\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md154}{}\doxysubsubsection{\texorpdfstring{Q\+: How can I use this on a machine without mouse, keyboard or screen? (input share, remote display)}{Q\+: How can I use this on a machine without mouse, keyboard or screen? (input share, remote display)}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md154}

\begin{DoxyItemize}
\item You can share your computer mouse seamlessly with your console/tablet/phone using solutions such as \href{https://symless.com/synergy}{\texttt{ Synergy}} This is the preferred solution for developer productivity. In particular, the \href{https://github.com/symless/micro-synergy-client}{\texttt{ micro-\/synergy-\/client repository}} has simple and portable source code (\doxylink{u_synergy_8c}{u\+Synergy.\+c}/.h) for a small embeddable client that you can use on any platform to connect to your host computer, based on the Synergy 1.\+x protocol. Make sure you download the Synergy 1 server on your computer. Console SDK also sometimes provide equivalent tooling or wrapper for Synergy-\/like protocols.
\item Game console users\+: consider emulating a mouse cursor with Dual\+Shock4 touch pad or a spare analog stick as a mouse-\/emulation fallback.
\item You may also use a third party solution such as \href{https://github.com/sammyfreg/netImgui}{\texttt{ net\+Imgui}}, \href{https://github.com/JordiRos/remoteimgui}{\texttt{ Remote Im\+Gui}} or \href{https://github.com/ggerganov/imgui-ws}{\texttt{ imgui-\/ws}} which sends the vertices to render over the local network, allowing you to use Dear \doxylink{namespace_im_gui}{Im\+Gui} even on a screen-\/less machine. See \href{https://github.com/ocornut/imgui/wiki}{\texttt{ Wiki}} index for most details.
\item For touch inputs, you can increase the hit box of widgets (via the {\ttfamily style.\+Touch\+Padding} setting) to accommodate for the lack of precision of touch inputs, but it is recommended you use a mouse or gamepad to allow optimizing for screen real-\/estate and precision.
\end{DoxyItemize}\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md155}{}\doxysubsubsubsubsection{\texorpdfstring{\protect\doxysectlink{index}{Return to Index}{0}}{{Return to Index}}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md155}

\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md157}{}\doxysubsubsection{\texorpdfstring{Q\+: I integrated Dear Im\+Gui in my engine and little squares are showing instead of text...}{Q\+: I integrated Dear Im\+Gui in my engine and little squares are showing instead of text...}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md157}
Your renderer backend is not using the font texture correctly or it hasn\textquotesingle{}t been uploaded to the GPU.
\begin{DoxyItemize}
\item If this happens using the standard backends\+: A) have you modified the font atlas after {\ttfamily Im\+Gui\+\_\+\+Impl\+XXX\+\_\+\+New\+Frame()}? B) maybe the texture failed to upload, which {\bfseries{can if your texture atlas is too big}}. Also see \href{https://github.com/ocornut/imgui/blob/master/docs/FONTS.md}{\texttt{ docs/\+FONTS.\+md}}.
\item If this happens with a custom backend\+: make sure you have uploaded the font texture to the GPU, that all shaders are rendering states are setup properly (e.\+g. texture is bound). Compare your code to existing backends and use a graphics debugger such as \href{https://renderdoc.org}{\texttt{ Render\+Doc}} to debug your rendering states.
\end{DoxyItemize}\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md158}{}\doxysubsubsubsubsection{\texorpdfstring{\protect\doxysectlink{index}{Return to Index}{0}}{{Return to Index}}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md158}

A primer on labels and the ID Stack...

Dear \doxylink{namespace_im_gui}{Im\+Gui} internally needs to uniquely identify UI elements. Elements that are typically not clickable (such as calls to the Text functions) don\textquotesingle{}t need an ID. Interactive widgets (such as calls to Button buttons) need a unique ID.

{\bfseries{Unique IDs are used internally to track active widgets and occasionally associate state to widgets.~\newline
 Unique IDs are implicitly built from the hash of multiple elements that identify the "{}path"{} to the UI element.}}

Since Dear \doxylink{namespace_im_gui}{Im\+Gui} 1.\+85, you can use {\ttfamily Demo\texorpdfstring{$>$}{>}Tools\texorpdfstring{$>$}{>}ID Stack Tool} or call {\ttfamily \doxylink{namespace_im_gui_ab4a87bf4a874a8e5872b60b4a9b1d99e}{Im\+Gui\+::\+Show\+IDStack\+Tool\+Window()}}. The tool display intermediate values leading to the creation of a unique ID, making things easier to debug and understand.

{\bfseries{Please read documentations or tutorials on your graphics API to understand how to display textures on the screen before moving onward.}}

Long explanation\+:
\begin{DoxyItemize}
\item Dear \doxylink{namespace_im_gui}{Im\+Gui}\textquotesingle{}s job is to create "{}meshes"{}, defined in a renderer-\/agnostic format made of draw commands and vertices. At the end of the frame, those meshes (\doxylink{struct_im_draw_list}{Im\+Draw\+List}) will be displayed by your rendering function. They are made up of textured polygons and the code to render them is generally fairly short (a few dozen lines). In the examples/ folder, we provide functions for popular graphics APIs (Open\+GL, DirectX, etc.).
\item Each rendering function decides on a data type to represent "{}textures"{}. The concept of what is a "{}texture"{} is entirely tied to your underlying engine/graphics API. We carry the information to identify a "{}texture"{} in the Im\+Texture\+ID type. Im\+Texture\+ID default to \doxylink{imgui_8h_a6875b9978368bbd5cf7e9b967e22eea4}{Im\+U64} aka 8 bytes worth of data\+: just enough to store one pointer or integer of your choice. Dear \doxylink{namespace_im_gui}{Im\+Gui} doesn\textquotesingle{}t know or understand what you are storing in Im\+Texture\+ID, it merely passes Im\+Texture\+ID values until they reach your rendering function.

\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md188}{}\doxysection{\texorpdfstring{Q\&A\+: Fonts, Text}{Q\&A\+: Fonts, Text}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md188}
\hypertarget{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md189}{}\doxysubsubsection{\texorpdfstring{Q\+: How should I handle DPI in my application?}{Q\+: How should I handle DPI in my application?}}\label{md_runtime_2imgui_2docs_2_f_a_q_autotoc_md189}
The short answer is\+: obtain the desired DPI scale, load your fonts resized with that scale (always round down fonts size to the nearest integer), and scale your Style structure accordingly using {\ttfamily style.\+Scale\+All\+Sizes()}.

Your application may want to detect DPI change and reload the fonts and reset style between frames.

Your UI code should avoid using hardcoded constants for size and positioning. Prefer to express values as multiple of reference values such as {\ttfamily \doxylink{namespace_im_gui_ac592ae8e0aa4a1a06502ec7872bc8da8}{Im\+Gui\+::\+Get\+Font\+Size()}} or {\ttfamily \doxylink{namespace_im_gui_ae560f17cf3262017cee888d1eb77f294}{Im\+Gui\+::\+Get\+Frame\+Height()}}. So e.\+g. instead of seeing a hardcoded height of 500 for a given item/window, you may want to use {\ttfamily 30\texorpdfstring{$\ast$}{*}\+Im\+Gui\+::\+Get\+Font\+Size()} instead.

Down the line Dear \doxylink{namespace_im_gui}{Im\+Gui} will provide a variety of standardized reference values to facilitate using this.

Applications in the {\ttfamily examples/} folder are not DPI aware partly because they are unable to load a custom font from the file-\/system (may change that in the future).

The reason DPI is not auto-\/magically solved in stock examples is that we don\textquotesingle{}t yet have a satisfying solution for the "{}multi-\/dpi"{} problem (using the {\ttfamily docking} branch\+: when multiple viewport windows are over multiple monitors using different DPI scales). The current way to handle this on the application side is\+:
\begin{DoxyItemize}
\item Create and maintain one font atlas per active DPI scale (e.\+g. by iterating {\ttfamily platform\+\_\+io.\+Monitors\mbox{[}\mbox{]}} before {\ttfamily New\+Frame()}).
\item Hook {\ttfamily platform\+\_\+io.\+On\+Changed\+Viewport()} to detect when a {\ttfamily Begin()} call makes a Dear \doxylink{namespace_im_gui}{Im\+Gui} window change monitor (and therefore DPI).

\end{DoxyItemize}

This approach is relatively easy and functional but comes with two issues\+:
\begin{DoxyItemize}
\item It\textquotesingle{}s not possibly to reliably size or position a window ahead of {\ttfamily Begin()} without knowing on which monitor it\textquotesingle{}ll land.
\item Style override may be lost during the {\ttfamily Begin()} call crossing monitor boundaries. You may need to do some custom scaling mumbo-\/jumbo if you want your {\ttfamily On\+Changed\+Viewport()} handler to preserve style overrides.
\end{DoxyItemize}

Please note that if you are not using multi-\/viewports with multi-\/monitors using different DPI scales, you can ignore that and use the simpler technique recommended at the top.

On Windows, in addition to scaling the font size (make sure to round to an integer) and using {\ttfamily style.\+Scale\+All\+Sizes()}, you will need to inform Windows that your application is DPI aware. \item For GLFW\+: this is done automatically.
\item For other Windows projects with other backends, or wrapper projects\+:

\item We provide a {\ttfamily \doxylink{imgui__impl__win32_8cpp_afda525ac1c8a2b4e357f19488f661014}{Im\+Gui\+\_\+\+Impl\+Win32\+\_\+\+Enable\+Dpi\+Awareness()}} helper method in the Win32 backend.

\end{DoxyItemize}
