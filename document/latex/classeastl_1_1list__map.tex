\subsubsection*{template$<$typename Key, typename T, typename \mbox{\hyperlink{namespaceeastl_ae3d33fc51c4c15a85aee95335a5d336d}{Compare}} = eastl\+::less$<$\+Key$>$, typename Allocator = EASTLAllocator\+Type$>$\newline
class eastl\+::list\+\_\+map$<$ Key, T, Compare, Allocator $>$}
\doxylink{classeastl_1_1list__map}{list\+\_\+map}

Implements a map like container, which also provides functionality similar to a list.

Note\+: Like a map, keys must still be unique. As such, \doxylink{classeastl_1_1list__map_ac4c4db6d53ee4216e11f8b3ab9b40133}{push\+\_\+back()} and \doxylink{classeastl_1_1list__map_a571b6526591273ed57a472b66ca9fc1e}{push\+\_\+front()} operations return a bool indicating success, or failure if the entry\textquotesingle{}s key is already in use.

\doxylink{classeastl_1_1list__map}{list\+\_\+map} is designed to improve performance for situations commonly implemented as\+: A map, which must be iterated over to find the oldest entry, or purge expired entries. A list, which must be iterated over to remove a player\textquotesingle{}s record when they sign off.

\doxylink{classeastl_1_1list__map}{list\+\_\+map} requires a little more memory per node than either a list or map alone, and many of \doxylink{classeastl_1_1list__map}{list\+\_\+map}\textquotesingle{}s functions have a higher operational cost (CPU time) than their counterparts in list and map. However, as the node count increases, \doxylink{classeastl_1_1list__map}{list\+\_\+map} quickly outperforms either a list or a map when find \mbox{[}by-\/index\mbox{]} and front/back type operations are required.

In essence, \doxylink{classeastl_1_1list__map}{list\+\_\+map} avoids O(n) iterations at the expense of additional costs to quick (O(1) and O(log n) operations\+: \doxylink{classeastl_1_1list__map_a571b6526591273ed57a472b66ca9fc1e}{push\+\_\+front()}, \doxylink{classeastl_1_1list__map_ac4c4db6d53ee4216e11f8b3ab9b40133}{push\+\_\+back()}, \doxylink{classeastl_1_1list__map_aa148529a57c3839c3b2e5f213ff3f403}{pop\+\_\+front()} and \doxylink{classeastl_1_1list__map_a079f1c5f4a8f3ab6f0efc8339ed961a6}{pop\+\_\+back()} have O(log n) operation time, similar to \doxylink{classeastl_1_1map_acb6a39a1678fb88e7d17142309df6bfe}{map\+::insert()}, rather than O(1) time like a list, however, \doxylink{classeastl_1_1list__map_a499763c8a2ce5620aac08510be028eb3}{front()} and \doxylink{classeastl_1_1list__map_a8e62505078c36ae3d2c50f986deaf7cc}{back()} maintain O(1) operation time.

As a canonical example, consider a large backlog of player group invites, which are removed when either\+: The invitation times out -\/ in main loop\+: while( !list\+Map.\doxylink{classeastl_1_1list__map_a4337adb7c08961ef5224173dca2fc621}{empty()} \&\& list\+Map.\+front().Is\+Expired() ) \{ list\+Map.\+pop\+\_\+front(); \} The player rejects the outstanding invitation -\/ on rejection\+: iter = list\+Map.\+find(player\+Id); if (iter != list\+Map.\+end()) \{ list\+Map.\+erase(iter); \}

For a similar example, consider a high volume pending request container which must\+: Time out old requests (similar to invites timing out above) Remove requests once they\textquotesingle{}ve been handled (similar to rejecting invites above)

For such usage patterns, the performance benefits of \doxylink{classeastl_1_1list__map}{list\+\_\+map} become dramatic with common O(n) operations once the node count rises to hundreds or more.

When high performance is a priority, Containers with thousands of nodes or more can quickly result in unacceptable performance when executing even infrequenty O(n) operations.

In order to maintain strong performance, avoid iterating over \doxylink{classeastl_1_1list__map}{list\+\_\+map} whenever possible.

find\+\_\+as In order to support the ability to have a tree of strings but be able to do efficiently lookups via char pointers (i.\+e. so they aren\textquotesingle{}t converted to string objects), we provide the find\+\_\+as function. This function allows you to do a find with a key of a type other than the tree\textquotesingle{}s key type. See the find\+\_\+as function for more documentation on this.

Pool allocation If you want to make a custom memory pool for a \doxylink{classeastl_1_1list__map}{list\+\_\+map} container, your pool needs to contain items of type \doxylink{classeastl_1_1list__map_a00455c9d2be22b8dc9edb2faa02825e3}{list\+\_\+map\+::node\+\_\+type}. So if you have a memory pool that has a constructor that takes the size of pool items and the count of pool items, you would do this (assuming that Memory\+Pool implements the Allocator interface)\+: typedef \doxylink{classeastl_1_1list__map}{list\+\_\+map}\texorpdfstring{$<$}{<}Widget, int, \doxylink{structeastl_1_1less}{less$<$\+Widget$>$}, Memory\+Pool\texorpdfstring{$>$}{>} Widget\+Map; // Delare your Widget\+Map type. Memory\+Pool my\+Pool(sizeof(\+Widget\+Map\+::node\+\_\+type), 100); // Make a pool of 100 Widget nodes. Widget\+Map my\+Map(\&my\+Pool); // Create a map that uses the pool.
