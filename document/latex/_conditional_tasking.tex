\doxysection{Conditional Tasking}
\hypertarget{_conditional_tasking}{}\label{_conditional_tasking}\index{Conditional Tasking@{Conditional Tasking}}
Parallel workloads often require making control-\/flow decisions across dependent tasks. Taskflow supports a very efficient interface of conditional tasking for users to implement general control flow such as dynamic flow, cycles and conditionals that are otherwise difficult to do with existing frameworks.\hypertarget{_conditional_tasking_CreateAConditionTask}{}\doxysubsection{\texorpdfstring{Create a Condition Task}{Create a Condition Task}}\label{_conditional_tasking_CreateAConditionTask}

When you submit a task to an executor, the scheduler starts with tasks of {\itshape zero dependencies} (both zero strong and weak dependencies) and continues to execute successive tasks whenever their {\itshape strong dependencies} are met. However, the scheduler skips this rule when executing a condition task and jumps directly to its successors indexed by the return value.

Each task has an {\itshape atomic} join counter to keep track of strong dependencies that are met at runtime. When a task completes, the join counter is restored to the task\textquotesingle{}s strong dependency number in the graph, such that the subsequent execution can reuse the counter again.\hypertarget{_conditional_tasking_TaskLevelSchedulingExample}{}\doxysubsubsection{\texorpdfstring{Example}{Example}}\label{_conditional_tasking_TaskLevelSchedulingExample}
Let\textquotesingle{}s take a look at an example to understand how task-\/level scheduling works. Suppose we have the following taskflow of one condition task {\ttfamily cond} that forms a loop to itself on returning {\ttfamily 0} and moves on to {\ttfamily stop} on returning {\ttfamily 1}\+:

The scheduler starts with {\ttfamily init} task because it has no dependencies (both strong and weak dependencies). Then, the scheduler moves on to the condition task {\ttfamily cond}. If {\ttfamily cond} returns {\ttfamily 0}, the scheduler enqueues {\ttfamily cond} and runs it again. If {\ttfamily cond} returns {\ttfamily 1}, the scheduler enqueues {\ttfamily stop} and then moves on.\hypertarget{_conditional_tasking_AvoidCommonPitfalls}{}\doxysubsection{\texorpdfstring{Avoid Common Pitfalls}{Avoid Common Pitfalls}}\label{_conditional_tasking_AvoidCommonPitfalls}
Condition tasks are handy in creating dynamic and cyclic control flows, but they are also easy to make mistakes. It is your responsibility to ensure a taskflow is properly conditioned. Top things to avoid include {\itshape no source tasks} to start with and {\itshape task race}. The figure below shows common pitfalls and their remedies.

In the {\ttfamily error1} scenario, there is no source task for the scheduler to start with, and the simplest fix is to add a task {\ttfamily S} that has no dependencies. In the {\ttfamily error2} scenario, {\ttfamily D} might be scheduled twice by {\ttfamily E} through the strong dependency and {\ttfamily C} through the weak dependency (on returning {\ttfamily 1}). To fix this problem, you can add an auxiliary task {\ttfamily D-\/aux} to break the mixed use of strong dependency and weak dependency. In the risky scenario, task {\ttfamily X} may be raced by {\ttfamily M} and {\ttfamily P} if {\ttfamily M} returns {\ttfamily 0} and P returns {\ttfamily 1}.
