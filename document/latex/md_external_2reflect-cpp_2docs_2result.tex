In this documentation, we have, on several occasions, read a JSON string like this\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keyword}{const}\ \textcolor{keyword}{auto}\ homer\ =\ \mbox{\hyperlink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl::json::read<Person>}}(json\_string).value();}

\end{DoxyCode}


However, so far we have not elaborated on why {\ttfamily .\doxylink{imgui__impl__opengl3__loader_8h_abe8e049f756b5ba547bda825af81b645}{value()}} is necessary. Why can\textquotesingle{}t we just do this?


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keyword}{const}\ \textcolor{keyword}{auto}\ homer\ =\ \mbox{\hyperlink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl::json::read<Person>}}(json\_string);}

\end{DoxyCode}


The reason is that parsing a JSON string is something that might fail. We do not know much about the string that put into there. It might not be a valid JSON. And even if it is a valid JSON, then it might not conform to the assumptions that we have made about it and laid out in the type system in the form of various structs and containers.

{\ttfamily \doxylink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl\+::json\+::read}} handles this by returning an {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}} type. {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}} is a way of error handling without exceptions. This is needed for two reasons\+:

1) Exceptions are controversial, because they add another, hidden control path to your program that it hard to follow or predict. For this reasons, and others, the Google C++ style guide \href{https://google.github.io/styleguide/cppguide.html\#Exceptions}{\texttt{ disallows exceptions altogether}}. Modern programming languages like Go and Rust don\textquotesingle{}t even support them in the first place. The C++ standards committee has recognized this and introduced \href{https://en.cppreference.com/w/cpp/utility/expected}{\texttt{ std\+::expected}} in C++-\/23. However, reflect-\/cpp is a library for C++-\/20 and until that changes, we need to implement our own result type.

2) In some cases, you want to signal to the parser that is fine if some things go wrong. {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}} is the way to do that.

If you do not care about these objections to exceptions, you can just call {\ttfamily \doxylink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl\+::json\+::read}\texorpdfstring{$<$}{<}...\texorpdfstring{$>$}{>}(json\+\_\+string).\doxylink{imgui__impl__opengl3__loader_8h_abe8e049f756b5ba547bda825af81b645}{value()}}, which throws an exception, if the result type contains an error and be done with it.

{\bfseries{Note}}\+: If you want to use {\ttfamily std\+::expected} instead of our own Result type, you can pass {\ttfamily -\/DREFLECTCPP\+\_\+\+USE\+\_\+\+STD\+\_\+\+EXPECTED} during compilation. This requires C++-\/23 support and may not be supported on all C++ compilers.\hypertarget{md_external_2reflect-cpp_2docs_2result_autotoc_md620}{}\doxysection{\texorpdfstring{What is {\ttfamily rfl\+::\+Result}?}{What is {\ttfamily rfl\+::\+Result}?}}\label{md_external_2reflect-cpp_2docs_2result_autotoc_md620}
{\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}} is similar to {\ttfamily std\+::optional} it that it contains an object that may or may not be there. However, unlike {\ttfamily std\+::optional}, it does not simply contain {\ttfamily std\+::nullopt} if the operation has not been successful. Instead, it contains an {\ttfamily \doxylink{classrfl_1_1_error}{rfl\+::\+Error}} with a clear error message.

In laying out our structs an containers, about the the type system, we are making {\itshape requirements} about the JSON input we are expecting from the outside world. {\ttfamily \doxylink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl\+::json\+::read}\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}(...)} checks whether these requirements are met. If the requirements are met, {\ttfamily \doxylink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl\+::json\+::read}\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}} returns T wrapped inside {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}}. If they are not met, it returns {\ttfamily \doxylink{classrfl_1_1_error}{rfl\+::\+Error}} containing an error message explaining what went wrong.\hypertarget{md_external_2reflect-cpp_2docs_2result_autotoc_md621}{}\doxysection{\texorpdfstring{Using {\ttfamily rfl\+::\+Result} for parsing}{Using {\ttfamily rfl\+::\+Result} for parsing}}\label{md_external_2reflect-cpp_2docs_2result_autotoc_md621}
Suppose you have a vector with 1000 user-\/supplied configurations\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keyword}{const}\ \textcolor{keyword}{auto}\ configs\ =\ \mbox{\hyperlink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl::json::read<std::vector<Config>}}>(json\_string);}

\end{DoxyCode}


Chances are that the user of your software has made as mistake in at least some of these configurations. If you set it up as shown above, {\ttfamily \doxylink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl\+::json\+::read}} will fail if one or more {\ttfamily Config} items contain an error. But sometimes you don\textquotesingle{}t want that. Sometimes you want your software to proceed with the remaining 999 configurations and gently point out to the user that some of the configurations were faulty.

The solution is to this instead\+:


\begin{DoxyCode}{0}
\DoxyCodeLine{\textcolor{keyword}{const}\ \textcolor{keyword}{auto}\ configs\ =\ \mbox{\hyperlink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl::json::read<std::vector<rfl::Result<Config>}}>>(json\_string);}

\end{DoxyCode}


This means {\ttfamily \doxylink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl\+::json\+::read}} is going to cut the end-\/users of your software some slack. But it also means that you, as the programmer, have to handle the fact that some of the configs might contain errors.

Below, we will show you how to do that.\hypertarget{md_external_2reflect-cpp_2docs_2result_autotoc_md622}{}\doxysection{\texorpdfstring{Monadic operations}{Monadic operations}}\label{md_external_2reflect-cpp_2docs_2result_autotoc_md622}
From the point of view of category theory, {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}} is both a functor and a monad (actually, all monads are functors).

This is reflected in its member functions {\ttfamily .transform(...)} and {\ttfamily .and\+\_\+then(...)}.

{\ttfamily .transform(...)} requires a function {\ttfamily F} of the type {\ttfamily T -\/\texorpdfstring{$>$}{>} U}. If the {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}} contains a type {\ttfamily T}, then {\ttfamily r.\+transform(f)} {\itshape will} return an {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}U\texorpdfstring{$>$}{>}} containing a type {\ttfamily U}, otherwise it will return an {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}U\texorpdfstring{$>$}{>}} containing an error.

{\ttfamily .and\+\_\+then(...)} requires a function {\ttfamily F} of the type {\ttfamily T -\/\texorpdfstring{$>$}{>} \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}U\texorpdfstring{$>$}{>}}. If the {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}} contains a type {\ttfamily T}, then {\ttfamily r.\+and\+\_\+then(f)} {\itshape might} return an {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}U\texorpdfstring{$>$}{>}} containing a type {\ttfamily U}, depending on whether {\ttfamily f} failed. If {\ttfamily r} already contained an error, then {\ttfamily r.\+and\+\_\+then(f)} will surely return an {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}U\texorpdfstring{$>$}{>}} containing an error.

This an operation that might fail (maybe the file doesn\textquotesingle{}t exist, maybe it isn\textquotesingle{}t readable, ...). Then we have to to parse its contents using {\ttfamily \doxylink{namespacerfl_1_1json_a905573518eca04a9c269d1bb32de7dd1}{rfl\+::json\+::read}\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}}. Again, this is an operation that might fail (might not be a proper JSON, might not conform to our requirements, ...). Therefore, we have to call {\ttfamily .and\+\_\+then(...)}.\hypertarget{md_external_2reflect-cpp_2docs_2result_autotoc_md623}{}\doxysection{\texorpdfstring{Retrieving values or errors}{Retrieving values or errors}}\label{md_external_2reflect-cpp_2docs_2result_autotoc_md623}
If you want the underlying value of type T, you can use {\ttfamily .\doxylink{imgui__impl__opengl3__loader_8h_abe8e049f756b5ba547bda825af81b645}{value(...)}}. Note that this throws an exception, if {\ttfamily \doxylink{classrfl_1_1_result}{rfl\+::\+Result}\texorpdfstring{$<$}{<}T\texorpdfstring{$>$}{>}} contains an error.

If you are 100\% sure that the result does not contain error, you can also call the operator {\ttfamily \texorpdfstring{$\ast$}{*}}. This will lead to undefined behavior, if you are wrong.

If you want the underlying value of type T or some kind of default value, you can call {\ttfamily .value\+\_\+or(your\+\_\+default\+\_\+value)}.

If you want to retrieve the error, you can call {\ttfamily .error()}, which will return a value of type {\ttfamily \doxylink{classrfl_1_1_error}{rfl\+::\+Error}}.
