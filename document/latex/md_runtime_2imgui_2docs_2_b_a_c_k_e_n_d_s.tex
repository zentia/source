Dear \doxylink{namespace_im_gui}{Im\+Gui} is highly portable and only requires a few things to run and render, typically\+:


\begin{DoxyItemize}
\item Required\+: providing mouse/keyboard inputs (fed into the {\ttfamily \doxylink{struct_im_gui_i_o}{Im\+Gui\+IO}} structure).
\item Required\+: uploading the font atlas texture into graphics memory.
\item Required\+: rendering indexed textured triangles with a clipping rectangle.
\end{DoxyItemize}

Extra features are opt-\/in, our backends try to support as many as possible\+:


\begin{DoxyItemize}
\item Optional\+: custom texture binding support.
\item Optional\+: clipboard support.
\item Optional\+: gamepad support.
\item Optional\+: mouse cursor shape support.
\item Optional\+: IME support.
\item Optional\+: multi-\/viewports support. etc.
\end{DoxyItemize}

This is essentially what each backend is doing + obligatory portability cruft. Using standard backends ensure you can get all those features including the ones that would be harder to implement on your side (e.\+g. multi-\/viewports support).

It is important to understand the difference between the core Dear \doxylink{namespace_im_gui}{Im\+Gui} library (files in the root folder) and the backends which we are describing here (backends/ folder).


\begin{DoxyItemize}
\item Some issues may only be backend or platform specific.
\item You should be able to write backends for pretty much any platform and any 3D graphics API. e.\+g. you can get creative and use software rendering or render remotely on a different machine.
\end{DoxyItemize}\hypertarget{md_runtime_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md116}{}\doxysubsubsection{\texorpdfstring{Standard backends}{Standard backends}}\label{md_runtime_2imgui_2docs_2_b_a_c_k_e_n_d_s_autotoc_md116}
{\bfseries{The \href{https://github.com/ocornut/imgui/blob/master/backends}{\texttt{ backends/}} folder contains backends for popular platforms/graphics API, which you can use in your application or engine to easily integrate Dear \doxylink{namespace_im_gui}{Im\+Gui}.}} Each backend is typically self-\/contained in a pair of files\+: imgui\+\_\+impl\+\_\+\+XXXX.\+cpp + imgui\+\_\+impl\+\_\+\+XXXX.\+h.


\begin{DoxyItemize}
\item The \textquotesingle{}Platform\textquotesingle{} backends are in charge of\+: mouse/keyboard/gamepad inputs, cursor shape, timing, and windowing.~\newline
 e.\+g. Windows (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_win32.cpp}{\texttt{ imgui\+\_\+impl\+\_\+win32.\+cpp}}), GLFW (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_glfw.cpp}{\texttt{ imgui\+\_\+impl\+\_\+glfw.\+cpp}}), SDL2 (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_sdl2.cpp}{\texttt{ imgui\+\_\+impl\+\_\+sdl2.\+cpp}}), etc.
\item The \textquotesingle{}Renderer\textquotesingle{} backends are in charge of\+: creating atlas texture, and rendering imgui draw data.~\newline
 e.\+g. Direct\+X11 (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_dx11.cpp}{\texttt{ imgui\+\_\+impl\+\_\+dx11.\+cpp}}), Open\+GL/\+Web\+GL (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_opengl3.cpp}{\texttt{ imgui\+\_\+impl\+\_\+opengl3.\+cpp}}), Vulkan (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_vulkan.cpp}{\texttt{ imgui\+\_\+impl\+\_\+vulkan.\+cpp}}), etc.
\item For some high-\/level frameworks, a single backend usually handles both \textquotesingle{}Platform\textquotesingle{} and \textquotesingle{}Renderer\textquotesingle{} parts.~\newline
 e.\+g. Allegro 5 (\href{https://github.com/ocornut/imgui/blob/master/backends/imgui_impl_allegro5.cpp}{\texttt{ imgui\+\_\+impl\+\_\+allegro5.\+cpp}}). If you end up creating a custom backend for your engine, you may want to do the same.
\end{DoxyItemize}

An application usually combines one Platform backend + one Renderer backend + main Dear \doxylink{namespace_im_gui}{Im\+Gui} sources. For example, the \href{https://github.com/ocornut/imgui/tree/master/examples/example_win32_directx11}{\texttt{ example\+\_\+win32\+\_\+directx11}} application combines \doxylink{imgui__impl__win32_8cpp}{imgui\+\_\+impl\+\_\+win32.\+cpp} + \doxylink{imgui__impl__dx11_8cpp}{imgui\+\_\+impl\+\_\+dx11.\+cpp}. There are 20+ examples in the \href{https://github.com/ocornut/imgui/blob/master/examples/}{\texttt{ examples/}} folder. See \href{https://github.com/ocornut/imgui/blob/master/docs/EXAMPLES.md}{\texttt{ EXAMPLES.\+MD}} for details.

You will likely be tempted to start by rewrite your own backend using your own custom/high-\/level facilities...~\newline
 Think twice!

If you are new to Dear \doxylink{namespace_im_gui}{Im\+Gui}, first try using the existing backends as-\/is. You will save lots of time integrating the library. You can LATER decide to rewrite yourself a custom backend if you really need to. In most situations, custom backends have fewer features and more bugs than the standard backends we provide. If you want portability, you can use multiple backends and choose between them either at compile time or at runtime.

{\bfseries{Example A}}\+: your engine is built over Windows + Direct\+X11 but you have your own high-\/level rendering system layered over Direct\+X11.~\newline
 Suggestion\+: try using \doxylink{imgui__impl__win32_8cpp}{imgui\+\_\+impl\+\_\+win32.\+cpp} + \doxylink{imgui__impl__dx11_8cpp}{imgui\+\_\+impl\+\_\+dx11.\+cpp} first. Once it works, if you really need it, you can replace the \doxylink{imgui__impl__dx11_8cpp}{imgui\+\_\+impl\+\_\+dx11.\+cpp} code with a custom renderer using your own rendering functions, and keep using the standard Win32 code etc.

{\bfseries{Example B}}\+: your engine runs on Windows, Mac, Linux and uses Direct\+X11, Metal, and Vulkan respectively.~\newline
 Suggestion\+: use multiple generic backends! Once it works, if you really need it, you can replace parts of backends with your own abstractions.

{\bfseries{Example C}}\+: your engine runs on platforms we can\textquotesingle{}t provide public backends for (e.\+g. PS4/\+PS5, Switch), and you have high-\/level systems everywhere.~\newline
 Suggestion\+: try using a non-\/portable backend first (e.\+g. win32 + underlying graphics API) to get your desktop builds working first. This will get you running faster and get your acquainted with how Dear \doxylink{namespace_im_gui}{Im\+Gui} works and is setup. You can then rewrite a custom backend using your own engine API...

Generally\+: It is unlikely you will add value to your project by creating your own backend.

Also\+: The \href{https://github.com/ocornut/imgui/wiki/Multi-Viewports}{\texttt{ multi-\/viewports feature}} of the \textquotesingle{}docking\textquotesingle{} branch allows Dear \doxylink{namespace_im_gui}{Im\+Gui} windows to be seamlessly detached from the main application window. This is achieved using an extra layer to the Platform and Renderer backends, which allows Dear \doxylink{namespace_im_gui}{Im\+Gui} to communicate platform-\/specific requests such as\+: "{}create an additional OS window"{}, "{}create a render context"{}, "{}get the OS position of this window"{} etc. See \textquotesingle{}\doxylink{struct_im_gui_platform_i_o}{Im\+Gui\+Platform\+IO}\textquotesingle{} for details. Supporting the multi-\/viewports feature correctly using 100\% of your own abstractions is more difficult than supporting single-\/viewport. If you decide to use unmodified imgui\+\_\+impl\+\_\+\+XXXX.\+cpp files, you can automatically benefit from improvements and fixes related to viewports and platform windows without extra work on your side.
