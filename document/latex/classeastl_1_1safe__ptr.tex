\subsubsection*{template$<$class T$>$\newline
class eastl\+::safe\+\_\+ptr$<$ T $>$}
\doxylink{classeastl_1_1safe__ptr}{safe\+\_\+ptr}

\doxylink{classeastl_1_1safe__ptr}{safe\+\_\+ptr} is an automatic, lightweight solution to the dangling pointer problem. This class is an alternative to \doxylink{classeastl_1_1weak__ptr}{weak\+\_\+ptr} which has the primary benefit of not allocating memory at the primary cost of being a tad slower and thread-\/unsafe.

During normal usage, safe\+\_\+ptr$<$\+T$>$ behaves exactly as a T\texorpdfstring{$\ast$}{*}. When the raw pointer referenced by the \doxylink{classeastl_1_1safe__ptr}{safe\+\_\+ptr} is deleted, all of the Safe\+Ptrs for the raw pointer are set to NULL.

This works by making the raw objects derive from the class \doxylink{classeastl_1_1safe__object}{safe\+\_\+object}, which maintains a linked-\/list of the Safe pointers that reference it. When a \doxylink{classeastl_1_1safe__object}{safe\+\_\+object} is destroyed, it walks its linked list, setting the object reference for each of its Safe\+Ptrs to NULL.

The overhead for this is light -\/ a single pointer is added to the size of the pointed to object, and a safe\+Ptr is the size of a raw pointer plus one list pointer.

This class is not thread-\/safe. In particular, manipulation of \doxylink{classeastl_1_1safe__ptr}{safe\+\_\+ptr} objects that refer to the same underlying object cannot be done safely from multiple threads. \doxylink{classeastl_1_1safe__ptr}{safe\+\_\+ptr} objects that are unrelated can be used safely from multiple threads.

Example usage\+: class Random\+Lifetime\+Object \+: public \doxylink{classeastl_1_1safe__object}{safe\+\_\+object} \{ public\+: Random\+Lifetime\+Object(); Method(); ... \};

safe\+\_\+ptr$<$\+Random\+Lifetime\+Object$>$ p\+Safe\+Ptr(new Random\+Lifetime\+Object); safe\+\_\+ptr$<$\+Random\+Lifetime\+Object$>$ p\+Safe\+Ptr\+Copy = p\+Safe\+Ptr;

p\+Safe\+Ptr-\/\texorpdfstring{$>$}{>}Method(); delete p\+Safe\+Ptr; At this point, p\+Safe\+Ptr\+Copy evaluates to NULL.
