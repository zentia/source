\subsubsection*{template$<$typename \+\_\+\+Scalar, class \+\_\+\+System$>$\newline
class Eigen\+::\+Euler\+Angles$<$ \+\_\+\+Scalar, \+\_\+\+System $>$}
Represents a rotation in a 3 dimensional space as three Euler angles. 

Euler rotation is a set of three rotation of three angles over three fixed axes, defined by the \doxylink{class_eigen_1_1_euler_system}{Euler\+System} given as a template parameter.

Here is how intrinsic Euler angles works\+:
\begin{DoxyItemize}
\item first, rotate the axes system over the alpha axis in angle alpha
\item then, rotate the axes system over the beta axis(which was rotated in the first stage) in angle beta
\item then, rotate the axes system over the gamma axis(which was rotated in the two stages above) in angle gamma
\end{DoxyItemize}

\begin{DoxyNote}{注解}
This class support only intrinsic Euler angles for simplicity, see \doxylink{class_eigen_1_1_euler_system}{Euler\+System} how to easily overcome this for extrinsic systems.
\end{DoxyNote}

It has been proved(see Wikipedia link below) that every rotation can be represented by Euler angles, but there is no singular representation (e.\+g. unlike rotation matrices). Therefore, you can convert from \doxylink{namespace_eigen}{Eigen} rotation and to them (including rotation matrices, which is not called "{}rotations"{} by \doxylink{namespace_eigen}{Eigen} design).

Euler angles usually used for\+:
\begin{DoxyItemize}
\item convenient human representation of rotation, especially in interactive GUI.
\item gimbal systems and robotics
\item efficient encoding(i.\+e. 3 floats only) of rotation for network protocols.
\end{DoxyItemize}

However, Euler angles are slow comparing to quaternion or matrices, because their unnatural math definition, although it\textquotesingle{}s simple for human. To overcome this, this class provide easy movement from the math friendly representation to the human friendly representation, and vise-\/versa.

All the user need to do is a safe simple C++ type conversion, and this class take care for the math. Additionally, some axes related computation is done in compile time.\hypertarget{class_eigen_1_1_euler_angles_autotoc_md1269}{}\doxysubsubsubsubsection{\texorpdfstring{Euler angles ranges in conversions}{Euler angles ranges in conversions}}\label{class_eigen_1_1_euler_angles_autotoc_md1269}
When converting some rotation to Euler angles, there are some ways you can guarantee the Euler angles ranges.\hypertarget{class_eigen_1_1_euler_angles_autotoc_md1270}{}\doxysubsubsubsubsection{\texorpdfstring{implicit ranges}{implicit ranges}}\label{class_eigen_1_1_euler_angles_autotoc_md1270}
When using implicit ranges, all angles are guarantee to be in the range \mbox{[}-\/PI, +PI\mbox{]}, unless you convert from some other Euler angles. In this case, the range is {\bfseries{undefined}} (might be even less than -\/PI or greater than +2\texorpdfstring{$\ast$}{*}\+PI). \begin{DoxySeeAlso}{参见}
