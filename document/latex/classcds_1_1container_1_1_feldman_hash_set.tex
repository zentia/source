\doxylink{struct_hash}{Hash} set based on multi-\/level array 

\label{classcds_1_1container_1_1_feldman_hash_set_cds_container_FeldmanHashSet_hp}%
\Hypertarget{classcds_1_1container_1_1_feldman_hash_set_cds_container_FeldmanHashSet_hp}%
 Source\+:
\begin{DoxyItemize}
\item \mbox{[}2013\mbox{]} Steven Feldman, Pierre La\+Borde, Damian Dechev "{}\+Concurrent Multi-\/level Arrays\+:          Wait-\/free Extensible Hash Maps"{}
\end{DoxyItemize}

\mbox{[}From the paper\mbox{]} The hardest problem encountered while developing a parallel hash map is how to perform a global resize, the process of redistributing the elements in a hash map that occurs when adding new buckets. The negative impact of blocking synchronization is multiplied during a global resize, because all threads will be forced to wait on the thread that is performing the involved process of resizing the hash map and redistributing the elements. {\ttfamily Feldman\+Hash\+Set} implementation avoids global resizes through new array allocation. By allowing concurrent expansion this structure is free from the overhead of an explicit resize, which facilitates concurrent operations.

The presented design includes dynamic hashing, the use of sub-\/arrays within the hash map data structure; which, in combination with {\bfseries{perfect hashing}}, means that each element has a unique final, as well as current, position. It is important to note that the perfect hash function required by our hash map is trivial to realize as any hash function that permutes the bits of the key is suitable. This is possible because of our approach to the hash function; we require that it produces hash values that are equal in size to that of the key. We know that if we expand the hash map a fixed number of times there can be no collision as duplicate keys are not provided for in the standard semantics of a hash map.

{\ttfamily Feldman\+Hash\+Set} is a multi-\/level array which has an internal structure similar to a tree\+:  The multi-\/level array differs from a tree in that each position on the tree could hold an array of nodes or a single node. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} position that holds a single node is a {\ttfamily data\+Node} which holds the hash value of a key and the value that is associated with that key; it is a simple struct holding two variables. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} {\ttfamily data\+Node} in the multi-\/level array could be marked. \doxylink{bench__gemm_8cpp_addc86e8508f14411ec98f521c520f875}{A} {\ttfamily marked\+Data\+Node} refers to a pointer to a {\ttfamily data\+Node} that has been bitmarked at the least significant bit (LSB) of the pointer to the node. This signifies that this {\ttfamily data\+Node} is contended. An expansion must occur at this node; any thread that sees this {\ttfamily marked\+Data\+Node} will try to replace it with an {\ttfamily array\+Node}; which is a position that holds an array of nodes. The pointer to an {\ttfamily array\+Node} is differentiated from that of a pointer to a {\ttfamily data\+Node} by a bitmark on the second-\/least significant bit.

{\ttfamily Feldman\+Hash\+Set} multi-\/level array is similar to a tree in that we keep a pointer to the root, which is a memory array called {\ttfamily head}. The length of the {\ttfamily head} memory array is unique, whereas every other {\ttfamily array\+Node} has a uniform length; a normal {\ttfamily array\+Node} has a fixed power-\/of-\/two length equal to the binary logarithm of a variable called {\ttfamily array\+Length}. The maximum depth of the tree, {\ttfamily max\+Depth}, is the maximum number of pointers that must be followed to reach any node. We define {\ttfamily current\+Depth} as the number of memory arrays that we need to traverse to reach the {\ttfamily array\+Node} on which we need to operate; this is initially one, because of {\ttfamily head}.

That approach to the structure of the hash set uses an extensible hashing scheme; {\bfseries{ the hash value is treated as a bit string}} and rehash incrementally.

\begin{DoxyNote}{注解}
Two important things you should keep in mind when you\textquotesingle{}re using {\ttfamily Feldman\+Hash\+Set}\+:
\begin{DoxyItemize}
\item all keys must be fixed-\/size. It means that you cannot use {\ttfamily std\+::string} as a key for {\ttfamily Feldman\+Hash\+Set}. Instead, for the strings you should use well-\/known hashing algorithms like \href{https://en.wikipedia.org/wiki/Secure_Hash_Algorithm}{\texttt{ SHA1, SHA2}}, \href{https://en.wikipedia.org/wiki/MurmurHash}{\texttt{ Murmur\+Hash}}, \href{https://en.wikipedia.org/wiki/CityHash}{\texttt{ City\+Hash}} or its successor \href{https://code.google.com/p/farmhash/}{\texttt{ Farm\+Hash}} and so on, which converts variable-\/length strings to fixed-\/length bit-\/strings, and use that hash as a key in {\ttfamily Feldman\+Hash\+Set}.
\item {\ttfamily Feldman\+Hash\+Set} uses a perfect hashing. It means that if two different keys, for example, of type {\ttfamily std\+::string}, have identical hash then you cannot insert both that keys in the set. {\ttfamily Feldman\+Hash\+Set} does not maintain the key, it maintains its fixed-\/size hash value.
\end{DoxyItemize}
\end{DoxyNote}
The set supports \doxylink{classcds_1_1container_1_1_feldman_hash_set_cds_container_FeldmanHashSet_iterators}{bidirectional thread-\/safe iterators}.

Bidirectional iterator 

\label{classcds_1_1container_1_1_feldman_hash_set_cds_container_FeldmanHashSet_iterators}%
\Hypertarget{classcds_1_1container_1_1_feldman_hash_set_cds_container_FeldmanHashSet_iterators}%
The set supports thread-\/safe iterators\+: you may iterate over the set in multi-\/threaded environment. It is guaranteed that the iterators will remain valid even if another thread deletes the node the iterator points to\+: Hazard Pointer embedded into the iterator object protects the node from physical reclamation.

\begin{DoxyNote}{注解}
Since the iterator object contains hazard pointer that is a thread-\/local resource, the iterator should not be passed to another thread.
\end{DoxyNote}
Each iterator object supports the following interface\+:
\begin{DoxyItemize}

Updates the element 

The operation performs inserting or replacing with lock-\/free manner.

If the {\ttfamily val} key not found in the set, then the new item created from {\ttfamily val} will be inserted into the set iff {\ttfamily b\+Insert} is {\ttfamily true}. Otherwise, if {\ttfamily val} is found, it is replaced with new item created from {\ttfamily val} and previous item is disposed. In both cases {\ttfamily func} functor is called.

\DoxyCodeLine{\};}

 where\+:
\begin{DoxyItemize}
\item {\ttfamily cur} -\/ current element
\item {\ttfamily prev} -\/ pointer to previous element with such hash. {\ttfamily prev} is {\ttfamily nullptr} if {\ttfamily cur} was just inserted.
\end{DoxyItemize}

The functor may change non-\/key fields of the {\ttfamily item}; however, {\ttfamily func} must guarantee that during changing no any other modifications could be made on this item by concurrent threads.

Returns {\ttfamily  std\+::pair$<$bool, bool$>$ } where {\ttfamily first} is {\ttfamily true} if operation is successful, i.\+e. the item has been inserted or updated, {\ttfamily second} is {\ttfamily true} if the new item has been added or {\ttfamily false} if the item with key equal to {\ttfamily val} already exists.
